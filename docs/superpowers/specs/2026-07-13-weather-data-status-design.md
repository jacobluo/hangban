# 天气数据状态展示设计

## 1. 背景

「数据覆盖与服务状态」页面当前只展示航班实时连接和航班数据源状态。天气雷达已作为独立的可选图层接入，但页面无法说明天气数据是否可获取、雷达帧是否仍属于当前天气，以及 RainViewer 服务是否处于降级状态。

天气雷达不进入航班 `SourceStatus[]`，也不应改变航班核心服务的总体健康判断。本设计在保留该领域边界的前提下，为状态页增加独立的天气数据情况。

## 2. 目标

- 打开状态页时主动检查天气雷达状态，即使地图上的天气雷达图层处于关闭状态。
- 独立展示天气供应商、服务状态、雷达帧时间、新鲜度说明和来源署名。
- 天气雷达不可用时，航班地图、航班数据源状态和总体健康判断保持不变。
- 现有「重新获取数据」操作同时刷新航班实时状态和天气雷达状态。
- 桌面端与手机端均能清楚区分加载、最新、延迟、历史缓存和不可用状态。

## 3. 不包含

- 不修改天气雷达后端契约、缓存期限或供应商适配器。
- 不把 RainViewer 加入航班数据源列表或 `SourceStatus[]`。
- 不新增雷达动画、历史查询、天气预报或多数据源切换。
- 不因打开状态页而自动开启地图天气雷达图层。
- 不新增天气服务 SLA、覆盖率百分比或无法从现有契约验证的区域覆盖结论。

## 4. 方案比较

### 4.1 采用方案：独立天气数据卡片

在航班数据源列表之后增加「天气数据」区域。状态页打开时复用天气雷达状态请求，但保持地图图层开关不变。

该方案能够明确区分航班实时位置与天气雷达两个数据域，并允许天气能力独立降级。界面可以使用天气专属的新鲜度语义，不需要把雷达帧强行映射为航班来源记录数或最近结果。

### 4.2 未采用：将 RainViewer 加入航班数据源列表

该方案可以复用现有数据源卡片，但会混淆航班实时位置与天气雷达语义，也会影响「正常航班来源」的分子和分母，因此不采用。

### 4.3 未采用：只在地图图层开启后展示天气状态

该方案请求较少，但状态页无法回答天气能力当前是否可用。用户已确认打开状态页时主动检查，因此不采用。

## 5. 信息结构

### 5.1 桌面端

- 保留现有总体状态摘要、航班数据源列表和覆盖说明。
- 在航班数据源列表下方增加独立的「天气数据」卡片。
- 卡片展示供应商、状态、雷达帧时间、有效性说明和可见署名。
- 天气卡片不可用时使用独立的降级样式，不改变页面顶部的航班总体健康状态。

### 5.2 手机端

- 天气数据使用独立的纵向卡片，位于航班数据源列表之后。
- 页面继续允许纵向滚动，不新增横向滚动。
- 状态、时间和说明按行折叠；来源链接保持可点击，触控区域不小于 `44 × 44 px`。

## 6. 状态与文案

天气数据卡片使用以下展示规则：

| 输入状态                                   | 展示状态           | 说明                                     |
| ------------------------------------------ | ------------------ | ---------------------------------------- |
| 请求进行中                                 | 「正在检查」       | 正在获取天气雷达状态，不影响航班状态展示 |
| `available: true`、`latest`                | 「最新」           | 雷达帧时间距当前时间不超过 15 分钟       |
| `available: true`、`delayed`               | 「数据延迟」       | 雷达帧时间距当前时间为 15 分钟至 2 小时  |
| `available: true`、`historical-cache`      | 「非当前天气」     | 展示缓存帧年龄，明确不能作为当前天气判断 |
| `available: false`、`DISABLED`             | 「服务未启用」     | 服务端配置未启用天气雷达                 |
| `available: false`、`UPSTREAM_UNAVAILABLE` | 「暂不可用」       | 当前无法从上游获得有效状态               |
| `available: false`、`NO_VALID_FRAME`       | 「暂无有效雷达帧」 | 上游响应中没有可使用的雷达帧             |
| `available: false`、`FRAME_EXPIRED`        | 「缓存已过期」     | 最近缓存帧已超过 24 小时有效范围         |
| 请求或响应解析失败                         | 「检查失败」       | 显示安全、可重试的提示，不暴露内部错误   |

可用状态展示后端返回的 `frameTime`，标签统一为「雷达帧时间」。页面不把该时间描述为所有区域的精确观测时间。

供应商固定显示为 RainViewer，并显示后端契约提供的 `Weather radar by RainViewer` 署名和链接。不可用响应没有署名字段时，前端仍使用已确认的 RainViewer 公共来源链接，不展示供应商主机或内部请求路径。

## 7. 总体健康与统计边界

- 页面顶部总体健康状态继续只由航班实时连接和航班 `SourceStatus[]` 决定。
- 「正常来源数」继续只统计航班数据源，RainViewer 不进入分子或分母。
- 天气雷达异常只在天气数据卡片中显示，不将「全部服务正常」改为「部分服务降级」。
- 天气数据卡片明确标记为「可选能力」，避免访客把天气异常理解为航班数据中断。

## 8. 数据流与交互

`AppShell` 在以下任一条件成立时请求天气雷达状态：

- 地图天气雷达图层已开启。
- 「数据覆盖与服务状态」页面已打开。

前端复用同一个天气雷达状态实例，不为状态页建立第二套请求和状态机。状态页打开只扩大请求条件，不修改 `mapLayers.weatherRadar`，因此不会创建 MapLibre raster 图层。

`DataStatusPage` 接收天气雷达的状态、加载状态、错误和重试回调。现有「重新获取数据」操作依次触发航班实时重试和天气雷达重试；其中一个请求失败时，另一个请求仍可完成。

关闭状态页后，如果地图天气雷达图层仍关闭，不再因为状态页保持额外轮询。再次打开状态页时重新检查，避免把较早的检查结果继续当作当前服务状态。

## 9. 可访问性

- 状态不能只通过颜色表达，所有天气状态均显示文字标签和说明。
- 署名使用可访问的链接名称，并在新标签页打开时保留安全的 `rel` 属性。
- 加载状态使用现有页面的状态语义，不抢占焦点。
- 「重新获取数据」保持键盘可操作，并在请求期间提供忙碌状态。

## 10. Ardot 设计门

修改前端实现前，必须更新 Ardot `main_deep（16:1）` 中以下已确认页面：

- PC：`Deep / Desktop / Data Coverage and Status（16:1970）`。
- 手机：`Deep / Mobile / Data Coverage and Status（16:2032）`。

两个页面都需要补齐天气数据卡片，并至少核对「最新」「非当前天气」和「暂不可用」三种视觉状态。完成设计确认后，才能修改 Next.js 实现。

## 11. 测试与验收

### 11.1 组件测试

- 地图天气雷达图层关闭时，打开状态页仍主动请求天气雷达状态。
- 打开状态页不会开启 MapLibre 天气雷达图层。
- 「最新」「数据延迟」「非当前天气」和各不可用原因显示正确文案。
- 天气不可用不改变航班总体健康状态和正常来源数。
- 「重新获取数据」同时触发航班和天气重试。
- RainViewer 署名可访问。

### 11.2 E2E 与视觉检查

- PC `1440 × 900`：天气卡片位于航班数据源列表之后，页面信息层级清晰。
- 手机 `390 × 844`：天气卡片可纵向阅读且没有水平溢出。
- 雷达图层关闭时打开状态页，能够看到主动检查后的天气状态。
- 天气接口失败时，航班状态和返回地图流程仍正常工作。
