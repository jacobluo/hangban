# 浏览器本地时间展示设计

## 1. 目标

统一「航迹」前端可见时间。服务端、数据库、缓存和实时事件继续使用带时区的 ISO 8601 UTC；浏览器按当前系统时区格式化时间，并显示 UTC 偏移，例如 `23:20 GMT+8`。

本设计只修改展示层，不迁移数据，不改变排序、新鲜度、缓存有效期、航迹回看范围或航班状态计算。

## 2. 已确认范围

- 航班摘要和完整详情中的最后观测时间。
- 航线探索中的更新时间。
- 顶部实时状态和数据覆盖状态页中的最后成功时间。
- 天气数据卡片和天气雷达图例中的雷达帧时间。
- 航迹回看控制中的时间刻度。
- 桌面端和手机端使用同一格式规则。

首期不包含计划起飞、计划到达等机场当地时间。未来获得可靠航班时刻后，机场当地时间应使用机场对应的 IANA 时区，不应复用浏览器时区。

## 3. 展示格式

提供两种格式：

| 类型     | 示例                       | 用途                         |
| -------- | -------------------------- | ---------------------------- |
| 紧凑时间 | `23:20 GMT+8`              | 摘要、状态入口、窄屏和图例   |
| 完整时间 | `2026/7/13 23:20:00 GMT+8` | 详情页、状态页和天气数据卡片 |

可见时间使用浏览器默认 locale 的数字规则，并强制显示 `timeZoneName: 'shortOffset'`。中文环境下日期顺序保持年、月、日。原始 UTC 写入 `<time dateTime="...">`，并作为 `title` 提示，例如 `UTC：2026-07-13 15:20:00`。

无效或缺失时间继续显示现有空状态，不把解析失败转换为当前时间。

## 4. 前端模块

新增统一时间模块：

- `formatBrowserTime(value, options)`：纯格式化函数，支持显式 `timeZone`，便于确定性测试。
- `BrowserTime`：统一渲染 `<time>`、可见文本、`dateTime` 和 UTC 提示。
- `useBrowserTimeReady`：确保服务端渲染与首次 hydration 使用同一占位，浏览器挂载后再读取系统时区。

业务组件不得继续通过 `slice()` 或 `toISOString()` 生成可见 UTC 时间。时间差、新鲜度和排序仍使用 `Date.parse()` 或毫秒时间戳。

## 5. Hydration 与时区变化

Next.js 可能在 UTC 服务器上预渲染客户端组件，而浏览器位于其他时区。`BrowserTime` 在服务端和首次客户端渲染时显示稳定占位 `—`，挂载后替换为浏览器本地时间，避免 React hydration 文本不一致。

浏览器在页面打开期间修改系统时区属于低频情况。首期在重新加载页面后生效，不增加持续轮询。

## 6. 可访问性

- 使用语义化 `<time>` 元素。
- `dateTime` 保留原始 ISO 8601 UTC。
- `title` 提供完整 UTC，便于跨时区核对。
- 不只依靠颜色区分时区或新鲜度。
- 紧凑布局不得省略 `GMT±N`。

## 7. Ardot 设计边界

在现有 `main_deep（16:1）` 中更新桌面端和手机端的时间示例：

- `08:20 UTC` 改为 `16:20 GMT+8`。
- `2026/7/13 06:20:00 UTC` 改为 `2026/7/13 14:20:00 GMT+8`。
- 保持卡片、表格和图例布局不变；只有文字宽度不足时才做最小宽度调整。

## 8. 测试与验收

- 格式化单元测试显式传入 `Asia/Shanghai`，验证 `GMT+8`、日期和秒级格式。
- 验证负偏移和夏令时由 `Intl.DateTimeFormat` 正确处理。
- 组件测试验证 hydration 占位、`dateTime` 和 UTC 提示。
- 现有新鲜度、缓存和排序测试保持不变。
- 搜索前端源码，禁止可见文案继续拼接 `UTC` 或截取 ISO 时间。
- Playwright 在 `Asia/Shanghai` 浏览器上下文中核对桌面端和手机端。
- 完成 1440 × 900 与 390 × 844 视觉核对。

## 9. 完成标准

- 所有面向访客的当前观测时间按浏览器时区展示并标明 UTC 偏移。
- 服务端契约和存储仍为 UTC。
- 首屏无 hydration 警告。
- 时间相关业务计算结果不变。
- Ardot、产品文档、自动化测试和实现保持一致。
