# 航空图表视觉深化设计规格

**日期：** 2026-07-12

**状态：** 已确认

**设计文件：** Ardot `hangban`（文件 ID `702710471706421`）

**基线页面：** `main`（页面 ID `0:1`）

**深化页面：** `main_deep`（页面 ID `16:1`，<https://ardot.tencent.com/file/702710471706421?node_id=16%3A1>）

## 1. 背景

「航迹」已经完成全球实时地图、统一搜索、航班详情、机场探索、航线探索、筛选、图层、数据状态和系统状态的桌面端与手机端设计，并已按 Ardot 原型实现首期界面。

当前方案具备完整的信息架构和数据真实性表达，但地图底图仍有较强的示意图观感，普通航班、选中航班、航线、机场和面板之间的视觉层级不够稳定。不同业务页面已经具备功能结构，但共用组件、状态语义和响应式规则仍需进一步统一。

本轮采用「浅色专业版」方向，在不改变首期产品边界的前提下，深化航空图表视觉语言和核心页面层级。参考设计只用于吸收地图视觉重心、选中对象聚焦和界面克制原则，不复制其中的账号、AI 规划、装饰性统计或虚构数据。

## 2. 目标

- 将当前界面从功能完整的浅色原型深化为可信、专业且具有辨识度的航空实时产品。
- 让地图成为首要视觉区域，并使选中航班、机场或航线在一秒内可识别。
- 统一全球实时、搜索、航班、机场、航线、筛选、数据状态和系统状态的视觉语言。
- 保持桌面端的信息密度，同时保证手机端主要任务完整可操作。
- 明确表达数据新鲜度、来源状态、覆盖度、降级状态和推断边界。
- 先在 Ardot 中完成并确认深化设计，再同步前端实现。

## 3. 非目标

本轮不包含：

- 深色主题、主题切换或独立的沉浸式全球观察模式。
- 账号、登录、头像、收藏同步或个性化设置。
- AI 航线规划、订票、值机、延误预测或专业空管能力。
- 无可靠数据支持的月度统计、全球在途总数、准点率、到港板或离港板。
- 为视觉效果虚构航班、机场、航线、计划时刻或数据覆盖。
- 修改实时数据适配层、统一领域模型、采集调度或持久化架构。

## 4. 设计原则

### 4.1 地图优先

地图承担主要视觉重量。未选中对象时，外围界面保持轻量；选中对象后，详情面板提供完整信息，但不得通过压缩地图或大幅移动地图中心破坏空间上下文。

### 4.2 单一焦点

普通航班使用低至中等对比度。当前选中对象使用橙色、尺寸、描边、标签和航线层级共同强调，不只依赖颜色区分。

### 4.3 数据可信

界面只展示统一模型已获得的字段。推断信息必须标记为「推断」或展示置信度。覆盖不足时不得用 `0` 表达「未获得记录」。

### 4.4 克制表达

页面使用航空图表式细描边、稳定网格、等宽关键数值和有限状态色。减少大面积浅蓝卡片、装饰性阴影和无意义统计。

### 4.5 响应式重组

手机端不是桌面端的等比缩小。列表、详情和筛选需要按照移动任务重新组织，并保持至少 44 × 44 px 的主要触控区域。

## 5. 视觉系统

### 5.1 语义颜色

在现有 `Aero Chart Tokens` 中补齐并统一以下语义变量。页面和组件不得直接依赖零散色值。

| 变量语义      | 基准值    | 用途                 |
| ------------- | --------- | -------------------- |
| Canvas        | `#F4F7FA` | 页面背景             |
| Ocean         | `#E9F0F6` | 地图海洋             |
| Land          | `#F8FAFC` | 地图陆地             |
| Surface       | `#FFFFFF` | 面板、抽屉和浮层     |
| Ink           | `#102A43` | 主文字               |
| Ink Muted     | `#627D98` | 辅助文字             |
| Border        | `#C9D5DF` | 普通边框             |
| Border Subtle | `#DDE6ED` | 分隔线               |
| Action        | `#0F62FE` | 操作和普通航班       |
| Selected      | `#FF6F3D` | 选中对象和已飞航迹   |
| Healthy       | `#159947` | 数据新鲜或服务正常   |
| Warning       | `#B77900` | 延迟、限流或部分降级 |
| Critical      | `#C93737` | 不可用或错误         |

最终色值允许在视觉核对时小幅调整，但语义边界不得改变：蓝色表示操作和普通实时对象，橙色表示当前选中对象，绿色表示正常，琥珀色表示警告，红色只表示明确错误。

### 5.2 间距、圆角与阴影

- 间距使用 `4 / 8 / 12 / 16 / 24 / 32 / 48`。
- 圆角使用 `4 / 8 / 12` 三档。
- 普通卡片通过细描边建立边界，不使用明显阴影。
- 阴影只用于浮层、抽屉和地图悬浮控制，并定义三个固定等级。

### 5.3 字体层级

- 页面标题：24–28 px，字重 600。
- 航班号：28–36 px，使用高辨识度窄体或等宽字体。
- 区块标题：14–16 px，字重 600。
- 正文：14 px。
- 辅助信息：12 px。
- 地图标签：11–12 px。
- 关键数值：16–20 px，使用等宽字体。
- 单位比数值小 2–4 px，并降低对比度。
- 中文正文行高不低于字号的 1.4 倍。

## 6. 全球实时地图

### 6.1 底图

- 海洋、陆地、海岸线、国界、城市和机场必须建立清晰的缩放级别层次。
- 海岸线比国界线高一级对比度，保证大陆轮廓完整。
- 国界在全球低缩放级别降低透明度，避免与航线争夺注意力。
- 城市和机场名称仅在相应缩放级别显示。
- 底图不得继续呈现不规则多边形草图观感；前端实现仍以 MapLibre 的真实地图数据为准。

### 6.2 航班与机场

- 普通航班标记为 16–18 px，使用 Action 色，默认透明度约 72%。
- 普通机场点为 4–6 px，使用低对比度蓝灰色。
- 密集区域必须通过缩放级别显示、聚合、抽样或碰撞隐藏控制噪声。
- 航班图标按航向旋转；过期航班降低透明度并显示最后更新时间。

### 6.3 选中航班

选中航班使用三层表达：

1. 24–28 px 的淡橙色外层光晕。
2. 白色底板或白色描边。
3. 使用 Selected 色、按航向旋转的飞机图标。

选中状态下，其他航班降低至约 35%–50% 透明度。选中标签展示航班号和必要的单项辅助信息，不在地图标签中复制完整详情。

### 6.4 选中航线

- 已飞部分使用 Selected 色实线。
- 未飞部分使用 Action 色虚线。
- 起终点使用 10 px 圆点和机场代码标签。
- 航线采用真实大圆插值，并正确处理日期变更线。
- 无法确认起终点时不得生成确定航线。

### 6.5 地图控制

- 桌面端控制尺寸为 40 × 40 px，手机端为 44 × 44 px。
- 定位、图层和缩放使用统一的白色或轻微透明背景、蓝灰描边和一致图标尺寸。
- 活跃状态使用浅蓝背景和深蓝图标。
- 手机端不常驻显示缩放加减按钮，保留手势缩放、定位和图层控制。

## 7. 顶部导航与实时状态

- 桌面端顶栏保持 64 px 高度。
- 一级导航保留「全球实时」「机场」「航线」。
- 当前入口使用 2 px 短指示线或轻量背景，不使用过长下划线。
- 右侧不展示无法验证的全球航班总数。
- 右侧改为统一实时状态入口，至少显示状态、最后更新时间和覆盖提示，例如「实时位置 · 8 秒前更新 · 部分覆盖」。
- 状态入口在桌面端打开数据覆盖与服务状态整页。
- 手机端将状态缩减为搜索框右侧状态点，点击后打开全高、可滚动的状态页面。

## 8. 搜索

### 8.1 桌面端

- 搜索框宽度为 400–440 px，高度为 48–52 px，与地图边缘保持 20–24 px 间距。
- 默认展示搜索图标、提示文字和快捷键；输入后显示清除按钮和结果类型。
- 搜索结果面板与搜索框等宽并贴合展开，最多展示 6–8 条结果。
- 结果按「航班」「机场」「城市」分组。
- 每条结果使用图标、主名称、代码或状态和辅助信息建立层级。
- 键盘选中项使用浅蓝背景和左侧 2 px 指示条。
- 匹配文字通过字重强调，不使用黄色背景。

### 8.2 手机端

- 搜索框位于安全区下方，高度为 52 px。
- 搜索后使用独立覆盖层或高位抽屉，避免与对象详情抽屉重叠。
- 键盘打开时，搜索结果优先使用可用高度，地图退居背景。

### 8.3 状态

- 无结果时保留关键字，并提供修改关键字和返回地图的动作。
- 网络错误时保留已输入内容，并提供重试。
- 搜索结果不得因当前地图视野限制而漏掉全球机场匹配。

## 9. 航班摘要与完整详情

### 9.1 桌面侧面板

右侧详情按以下顺序组织：

1. 航班身份：航班号、航空公司、机型和当前状态。
2. 航线关系：起终点、城市和飞行进度。
3. 飞行指标：高度、地速、航向和升降率。
4. 数据说明：更新时间、来源、覆盖度和推断说明。

面板宽度保持稳定。打开或关闭面板时，通过地图内容安全区调整相机，不得使选中航班离开可见区域。

### 9.2 信息表现

- 航班号是面板第一视觉焦点。
- 当前状态靠近航班号。
- 起终点区域使用白底、细描边和中央进度线，减少大面积浅蓝填充。
- 指标使用两列列表、细分隔线和等宽数值，不为每项指标创建同等视觉重量的色块。
- 来源和覆盖度形成独立可信度区域。
- 「查看完整详情」是面板唯一主操作。

### 9.3 完整详情

完整详情依次展示航班身份、实时状态、起终点关系、飞行指标、航空公司与机型补全信息、数据来源、覆盖度和推断说明。ADSBdb 补全字段必须与实时位置字段在语义上区分。

### 9.4 手机底部抽屉

- 抽屉包含收起、半展开和全展开三个稳定停靠点。
- 收起态展示航班号、状态、起终点和一项关键指标。
- 半展开态增加高度、速度和更新时间。
- 全展开态展示完整信息和数据说明。
- 主操作只在半展开和全展开状态显示。
- 地图控制始终位于抽屉最高位置以上。

## 10. 机场探索与机场详情

### 10.1 桌面布局

桌面端使用左侧机场列表、中间地图和按需出现的右侧详情。左侧建议宽度为 320 px；右侧建议宽度为 320–360 px。未选择机场时不显示右侧详情。

### 10.2 机场列表

- 明确区分「当前视野」与「全球搜索」。
- 当前视野模式显示地图范围内的机场数量。
- IATA 或 ICAO 代码是列表扫描锚点。
- 城市、国家、机场类型和距离作为辅助信息。
- 不展示到港、离港、登机口、延误或计划班次。

### 10.3 机场详情

- 顶部突出机场代码和中英文名称。
- 坐标、海拔和类型归入机场资料。
- 周边实时航班数量单独显示。
- 周边航班列表显示距离、方位、高度和更新时间。
- 固定说明「周边航班不等同于到港或离港班次」。
- 覆盖不足时显示「当前未获得记录」，不得显示为 `0`。

### 10.4 手机端

手机端采用机场列表、选中机场、底部详情的连续流程，不在同一抽屉中并列列表和完整详情。

## 11. 航线探索

- 航线页以起终点之间的空间关系为视觉中心。
- 起点和终点选择器保持结构对称，并提供交换按钮。
- 交换按钮只交换机场，不修改其他筛选。
- 起终点未完整选择时显示引导状态，不绘制虚假航线。
- 选择完成后，地图自动适配起终点和匹配航班。
- 大圆航线使用 Action 色，当前匹配航班使用 Selected 色，其他附近航班降低对比度。
- 航线概览展示距离、方向、当前匹配航班数、更新时间和覆盖度。
- 匹配航班可展示飞行进度，但必须说明匹配基于公开航班信息和实时位置归并。
- 不展示排班总数、准点率或未获得的计划数据。
- 手机端先完成起终点选择，再展开地图结果，输入控件不得长期占据地图上半屏。

## 12. 筛选与图层

- 图层控制「地图显示什么」，筛选控制「哪些航班进入结果」，两者不得混用。
- 桌面端使用约 320 px 的右侧轻量面板；手机端使用全高抽屉。
- 图层开关使用带图标的列表项。
- 高度范围使用双端滑块并提供数值输入。
- 航空公司使用可搜索多选。
- 数据质量使用「全部」「新鲜」「延迟」「过期」。
- 条件旁可显示应用后的剩余航班数。
- 即时生效和点击「应用」只能选择一种交互模型。建议采用即时生效，并保留「重置筛选」。
- 手机端在关闭前保留临时选择，主要触控区域不小于 44 × 44 px。

## 13. 数据覆盖与系统状态

### 13.1 数据覆盖页

顶部先给出用户可理解的结论，包括总体状态、实时位置是否继续更新、覆盖边界和最后成功更新时间。

每个数据源展示：

- 数据源名称。
- 最近请求结果。
- 最后成功时间。
- 最近成功记录数。
- 稳定错误类型。
- 是否正在使用缓存或上次成功数据。

页面底部解释「实时」的新鲜度定义、免费数据的覆盖边界、当前航班数的含义和可执行操作。

### 13.2 共用系统状态

所有核心页面必须覆盖：

- 首次加载。
- 正常更新。
- 搜索无结果。
- 当前视野无航班。
- 单个数据源不可用。
- 所有实时来源不可用。
- 网络断开和重连。
- 数据延迟和过期。
- 定位权限被拒绝。

状态颜色只作为辅助，必须同时使用文字、图标或结构差异。单个来源降级时继续展示有效数据；全部来源不可用时保留地图和机场静态信息，并隐藏不可用的实时指标。

## 14. 响应式规则

### 14.1 桌面端

- 主要验收视口为 1440 × 900。
- 顶栏固定为 64 px。
- 左侧探索区建议宽度为 320–360 px。
- 右侧详情区建议宽度为 340–380 px。
- 两侧面板同时出现时，地图有效宽度不得低于约 640 px。
- 宽度不足时优先折叠左侧列表，不压缩右侧详情。
- 底部回看条不得覆盖侧面板。

### 14.2 手机端

- 主要验收视口为 390 × 844。
- 搜索框、地图控制和底部抽屉互不遮挡。
- 航班、机场和航线共用抽屉容器，但使用不同内容结构。
- 手机端不常驻显示短时位置回看条。
- 复杂筛选使用全高抽屉。
- 必须处理安全区、软键盘、浏览器地址栏变化和横屏下的主要操作可达性。

## 15. 动效与无障碍

- 面板进入时间为 160–220 ms。
- 地图相机移动时间为 400–600 ms，并允许用户中途打断。
- 状态变化时间为 150–200 ms。
- 航班位置在数据更新周期内平滑插值，不瞬移。
- 底部抽屉使用自然减速，不使用弹跳。
- `prefers-reduced-motion` 下关闭位置插值和非必要过渡。
- 正文与背景至少达到 WCAG AA 对比度。
- 选中对象同时使用尺寸、描边和标签区分。
- 键盘可以完成搜索、选择结果、打开详情和关闭面板。
- 焦点态使用 2 px 品牌蓝外框，并与悬停态区分。

## 16. 共用组件

Ardot 深化应先建立或更新以下组件及其状态，再修改业务画板：

- 顶部导航和实时状态入口。
- 桌面搜索框、手机搜索框和搜索结果项。
- 航班标记：普通、悬停、选中、延迟和过期。
- 机场标记：普通、选中和低覆盖。
- 地图控制按钮。
- 航班身份头部和机场身份头部。
- 航线机场选择器。
- 指标单元。
- 数据来源与覆盖度区块。
- 列表项、筛选项和图层项。
- 状态横幅、空状态和错误状态。
- 桌面侧面板和手机底部抽屉。

组件实例不得通过复制后手工改色表达状态。选中、禁用、延迟、过期和错误必须使用统一语义变量和变体。

## 17. Ardot 执行要求

### 17.1 画板策略

- 不直接覆盖 `main` 页面中的当前已确认画板。
- 全部深化画板和新版共用组件放在独立的 `main_deep` 页面中。
- `main_deep` 页面按基础组件、全球实时、搜索、航班详情、机场、航线、筛选与图层、数据与系统状态分区组织。
- 深化版本通过评审后，再决定归档或替换旧画板。

### 17.2 顺序

1. 补齐 `Aero Chart Tokens`。
2. 建立共用组件和状态变体。
3. 深化桌面端与手机端全球实时地图。
4. 深化搜索结果、航班摘要和完整详情。
5. 深化机场探索和机场详情。
6. 深化航线探索。
7. 深化筛选、图层和数据覆盖页面。
8. 深化加载、空结果、断线、降级和过期状态。
9. 完成 1440 × 900 与 390 × 844 视觉核对。

每个阶段先完成 Ardot 设计与截图检查，再进入下一阶段。前端实现不得领先于未确认的深化原型。

## 18. 前端同步边界

Ardot 设计确认后，前端同步应：

- 复用现有 Next.js、React 和 MapLibre GL JS 架构。
- 保持浏览器只访问本项目 API 与 WebSocket。
- 使用真实地图数据和统一领域模型，不复刻 Ardot 中的示意几何或展示数字。
- 为新增或变更的用户可见行为补充组件测试和 E2E。
- 视觉变更同时检查 1440 × 900 与 390 × 844。
- 更新 `docs/product-design.md` 的视觉核对记录。

## 19. 验收条件

本轮设计满足以下条件后可进入实施计划：

- 所有核心页面均有桌面端和手机端深化画板。
- 全部画板使用统一语义颜色、间距、圆角和字体层级。
- 选中航班、机场和航线在地图上具有明确且一致的焦点。
- 普通对象、选中对象、延迟对象和过期对象可同时区分。
- 搜索、机场、航线、筛选、数据状态和系统状态具有统一组件语言。
- 页面未引入账号、AI 规划、虚构统计或无数据支持的航空信息。
- 数据新鲜度、覆盖度、降级状态和推断语义均有明确表达。
- 桌面端 1440 × 900 无关键内容遮挡、溢出或不可达操作。
- 手机端 390 × 844 无关键内容遮挡，主要触控区域不小于 44 × 44 px。
- Ardot 截图检查未发现重叠、裁切、文本溢出和明显对齐问题。
- 设计评审通过后，才创建前端同步实施计划。
