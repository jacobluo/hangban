# 机场与航班信息补全设计

## 1. 目标

在不引入登机口、延误原因、前序航班和账号体系的前提下，完成以下能力：

- 前端不再使用固定的 4 个演示机场。
- 地图和机场列表默认只展示当前视野内的机场。
- 全局搜索不受地图视野限制，支持机场代码、英文城市名、中文城市名、拼音和机场名称。
- 使用 ADSBdb 按需补充航司、机型、注册号和推断起终点。
- 机场或航班补充来源不可用时，实时位置展示保持可用。

## 2. 范围

### 2.1 包含

- OurAirports 每日同步。
- GeoNames 城市与多语言别名每周同步。
- 机场视野查询、全局搜索和代码详情查询。
- 世界视野机场抽稀和响应数量上限。
- ADSBdb 航空器、航司和呼号路线适配器。
- 航班元数据按需查询、缓存、负缓存和异步更新。
- 推断字段、来源和置信度表达。
- 机场探索页面的当前视野列表、搜索和继续加载交互。
- 桌面端与手机端原型和实现。

### 2.2 暂不包含

- 登机口、航站楼变更和行李转盘。
- 延误原因、官方延误预测和运行处置建议。
- 前序航班。
- 官方到港板和离港板。
- 值机、行程管理和旅客身份信息。
- 将 ADSBdb 路线匹配描述为官方飞行计划。

## 3. 数据来源

### 3.1 OurAirports

OurAirports 继续作为机场主数据来源，提供 IATA、ICAO、名称、城市、国家、坐标、海拔和机场类型。

- 默认每天同步一次。
- 同步失败时保留上一版本。
- 同步结果先校验，再原子替换。
- API 不直接返回完整机场文件。

### 3.2 GeoNames

GeoNames 用于补充城市的中文名称、英文名称、ASCII 名称和常用别名。

- 使用城市数据和 `alternateNamesV2` 数据。
- 默认每周同步一次。
- 仅保留机场搜索需要的城市、国家、坐标和语言别名。
- 保留 `zh`、`zh-CN`、`zh-Hans`、`zh-Hant`、`en` 和无语言 ASCII 名称。
- 按 GeoNames 许可要求在产品数据说明中标注来源。

机场与城市按以下顺序关联：

1. 国家代码一致且 `municipality` 与城市名称或别名规范化后完全匹配。
2. 无名称匹配时只保留 OurAirports 城市名称，不生成中文名称。

索引构建时预先完成名称关联，避免每次搜索重新计算。禁止用最近城市兜底，防止城市数据不完整时把相邻城市错误标注为同一城市。

### 3.3 ADSBdb

ADSBdb 用于按 ICAO24、注册号或呼号补充：

- 航空器注册号。
- 制造商和机型。
- 航空公司。
- 呼号对应的可能出发机场和到达机场。

ADSBdb 不作为实时位置来源。路线字段属于推断信息，必须带有 `inferredFields`、来源和较低置信度。

## 4. 统一模型

### 4.1 Airport

机场模型增加可选的搜索辅助字段：

```ts
type AirportSearchMetadata = {
  cityNames: Array<{
    language: string;
    value: string;
    preferred: boolean;
  }>;
  searchAliases: string[];
  geonamesId?: number;
  cityMatch?: {
    rule: 'name' | 'nearest';
    distanceKm: number;
  };
};
```

公开机场详情可以返回首选中文城市名，但内部别名索引不需要完整发送到浏览器。

### 4.2 Flight

沿用现有 `Flight` 字段，并补充字段级来源信息：

```ts
type FlightFieldSource = {
  field: 'airline' | 'aircraftType' | 'registration' | 'origin' | 'destination';
  providerId: string;
  observedAt: string;
  inferred: boolean;
  confidence: number;
};
```

位置、高度、速度和航向继续由 ADS-B 来源决定。ADSBdb 只填补缺失的元数据，不覆盖更新鲜或置信度更高的已有字段。

## 5. 机场查询设计

### 5.1 当前视野

新增接口：

```text
GET /api/v1/airports?bbox=west,south,east,north&zoom=number&cursor=string&limit=number
```

规则：

- `limit` 默认 100，最大 200。
- 中高缩放级别优先返回视野内全部大型和中型机场，再返回小型机场。
- 世界级缩放只返回大型机场，并按机场类型和稳定代码排序后截断。
- 返回 `nextCursor`，用于机场探索页面继续加载。
- bbox 查询只影响默认列表，不限制全局搜索。

### 5.2 全局搜索

扩展现有搜索接口：

```text
GET /api/v1/search?q=深圳&types=airport,flight&limit=20
```

机场匹配字段：

- IATA 和 ICAO。
- 英文机场名称。
- OurAirports 城市名称。
- GeoNames 中文、英文和 ASCII 城市别名。
- 规范化后的拼音或常用拉丁拼写。

排序顺序：

1. IATA 或 ICAO 完全匹配。
2. 航班号或注册号完全匹配。
3. 城市首选名称完全匹配。
4. 机场名称前缀匹配。
5. 城市或机场名称包含匹配。
6. 同分时大型机场优先，代码稳定排序。

搜索最多返回 20 条机场和 20 条航班。查询为空、超过长度限制或只包含无效字符时返回稳定错误码。

## 6. 航班补全流程

发现新的 ICAO24 或呼号后，补全协调器执行：

1. 查询本地缓存。
2. 缓存缺失时进入有界队列。
3. 按 ICAO24 查询航空器资料。
4. 按呼号查询航司和可能路线。
5. 校验响应并转换为统一候选字段。
6. 与当前航班合并。
7. 通过现有 `flight.upsert` 事件发送更新。

补全不得阻塞实时位置周期。单次补全失败不得把数据源整体标记为航班位置不可用。

默认策略：

- 并发数：4。
- 航空器资料缓存：24 小时。
- 呼号路线缓存：6 小时。
- 未找到或失败负缓存：5 分钟。
- 请求超时：5 秒。
- 单轮只处理仍存在于当前航班集合的任务。

## 7. 状态与降级

- OurAirports 同步失败：保留上一机场版本。
- GeoNames 同步失败：保留上一别名版本；英文和代码搜索继续可用。
- ADSBdb 失败：位置展示继续可用，缺失字段显示「未获得数据」。
- ADSBdb 路线冲突：不写入起终点，或保留置信度更高的现有值。
- 推断起终点必须加入 `inferredFields`，界面显示「根据公开路线信息推断」。
- 浏览器不得直接调用 GeoNames 或 ADSBdb，也不得持有第三方凭据。

## 8. 前端行为

### 8.1 地图

- 地图加载和 `moveend` 后请求当前 bbox 内机场。
- 快速连续移动时取消旧请求，只应用最新响应。
- 机场图层不再使用 testkit 数据。
- 世界视野仅绘制 API 返回的抽稀结果。

### 8.2 机场探索

- 默认列表跟随当前地图视野。
- 列表首屏数量由桌面端和手机端原型确定。
- 「继续加载」使用 bbox 查询游标，不切换为全球列表。
- 输入搜索词后切换为全球搜索结果；清空搜索后恢复当前视野列表。
- 无结果、加载中、网络失败和视野内无机场均有明确状态。

### 8.3 航班详情

- 实时位置先显示。
- 航司、机型、注册号和起终点可随后补齐。
- 字段补齐时不关闭详情、不重置地图选中态。
- 推断起终点显示语义说明。

机场探索列表、搜索状态和继续加载会改变现有 UI。按照仓库规则，必须先更新 Ardot 文件 `hangban` 的桌面端与手机端原型并完成确认，再修改前端实现。

## 9. 性能边界

- API 启动时构建机场代码索引、规范化文本索引和空间索引。
- 不向浏览器发送全部机场或全部搜索别名。
- bbox 查询最多返回 200 个机场。
- 全球搜索返回结果有明确上限。
- ADSBdb 查询有界并发、缓存和负缓存，避免按每次位置更新重复请求。
- 机场索引和航班补全缓存必须暴露数量指标，便于验证长期内存边界。

## 10. 测试与验收

### 10.1 机场

- 根同步命令生成通过契约校验的 OurAirports 和 GeoNames 数据。
- `深圳`、`shenzhen`、`SZX`、`ZGSZ` 均返回深圳宝安国际机场。
- `纽约` 和 `new york` 返回相关机场，并按规则排序。
- bbox 列表只包含当前视野机场。
- 世界视野不超过响应上限。
- 前端不再显示「已载入 4」。

### 10.2 航班

- 新航班先通过 ADS-B 位置进入地图。
- ADSBdb 返回后，航司、机型、注册号和推断路线通过 `flight.upsert` 更新。
- ADSBdb 超时或无结果时，航班位置仍持续更新。
- 推断起终点包含来源、置信度和 `inferredFields`。
- 同一 ICAO24 和呼号在缓存期内不重复请求。

### 10.3 门禁

- 单元测试覆盖同步、解析、索引、排序、缓存、负缓存和融合。
- API 集成测试覆盖 bbox、全局中文搜索和航班补全。
- Playwright 覆盖桌面端与手机端的视野机场、中文搜索、继续加载和航班异步补全。
- 完成后运行 `pnpm verify` 和 `pnpm format:check`。

## 11. 完成定义

- 前端机场数据完全来自 API，不再依赖 testkit 运行数据。
- 默认机场列表只显示当前视野，搜索覆盖全球。
- 中文、英文、代码和拼音搜索通过验收用例。
- 航班基础元数据可异步补齐，失败不影响实时位置。
- 推断字段不会被描述为官方确定事实。
- Ardot 桌面端和手机端原型先于 UI 实现完成并确认。
- 文档、测试、离线门禁和真实运行验证全部通过。

## 12. 已确认原型

Ardot 文件 `hangban` 已增加并确认以下原型，前端实现以这些节点为准：

- `12:2`：桌面端当前视野机场列表与继续加载。
- `12:210`：手机端当前视野机场列表与继续加载。
- `12:307`：桌面端全球中文机场搜索。
- `12:515`：手机端全球中文机场搜索。
- `12:612`：桌面端航班元数据补齐与推断路线说明。
- `12:685`：手机端航班元数据补齐与推断路线说明。
- `12:724`：机场与航班补全的加载、空结果和失败状态。
